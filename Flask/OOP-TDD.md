## Object Oriented Programming

> 객체지향 프로그래밍(Object Oriented Programming - OOP) 컴퓨터 프로그래밍의 패러다임 중 하나이다.

객체지향 프로그래밍이란, 프로그램을 명령어의 목록으로 보는 시각에서 벗어나 여러 개의 독립적인 단위, 즉 "객체"들의 모임으로 파악하고자 하는 것으로 각각의 객체는 메시지를 주고 받거나, 데이터를 처리할 수 있다. - wikipedia

정의만 살펴보면 기존에서 벗어났다는 점에서 과거의 단점을 해소했다고 보이는데, 계속해서 읽어보면 객체 지향 프로그래밍을 프로그램을 유연하고 변경이 용이하기 만들기 때문에 대규모 소프트웨어 개발에 많이 사용된다고 한다. 그럼 작은 규모의 개발에서는 사용하지 못하거나 않는가? 라고 생각하면 그건 또 아니라고 생각한다. 주로 대규모 소프트웨어 개발에 많이 사용되는 이유를 생각해보면 구조가 대규모인 경우에는 개발자가 한 명만 있는 것이 아니라, 여러 명 즉 다수의 인원이 개발에 참여하기 때문에 개개인 모두가 개발에 사용되는 모듈이나 코드에 대해서 이해할 수 있고 기여할 수 있는 구조여야한다. 그렇기에 대규모 시스템 개발에 더 적합할 뿐이지 작은 규모의 개발에는 사용되지 않는 것은 아니다.

이 외에도 다양한 장점들이 있는데 그 중 하나는 객체 지향 프로그래밍은 개발자로 하여금 코드를 이해하기 쉽게 하고 개발과 보수를 보다 간편하게 하고 직관적인 코드 분석을 가능하게 하는 장점을 갖는다. 하지만 개발을 진행하다보면 지나치게 코드를 객체화하는 경향이 있는데 이 경우에는 실제 세계(?)의 모습을 그대로 반영하지 못한다는 비판을 받는다고 되어 있다.



### 객체 지향적 설계 원칙

1. SRP (Single Responsibility Principle) : 단일 책임 원칙
2. OCP (Open-Closed Principle) : 개방-폐쇄 원칙
3. LSP (Liskov Substitution Principle) : 리스코프 치환 원칙
4. ISP (Interface Segregation Principle) : 인터페이스 분리 원칙
5. DIP (Dependency Inversion Principle) : 의존 역전 원칙

## TDD (Test-Driven Development)

> 테스트 주도형 개발 (Test-Driven Development - TDD) 란, 매우 짧은 개발 사이클의 반복에 의존하는 소프트웨어 개발 프로세스이다.

TDD란 테스트 주도형 개발로 개발자는 우선 요구되는 새로운 기능에 대한 자동화된 테스트케이스를 작성하고 해당 테스트를 통과하는 가장 간단한 코드를 작성한다. (음? 코드를 작성한 이후에 테스트 케이스를 작성하는 것이 아닌 반대였다니 !) 이렇게 일단 테스트를 통과하는 코드를 작성하고 상황에 맞게 리팩토링하는 과정을 거치는 것이다. 말 그대로 테스트가 코드 작성을 주도하는 개발방식을 말한다.

테스트 주도형 개발에선 새로운 기능 추가전 테스트를 가장 먼저 작성하는데, 테스트를 작성하기 위해서는 개발자는 해당 기능에 대한 요구사항과 명세를 분명히 이해하는 것이 중요하다. (요구사항 이해는 중요하다 !) 그 이유는 요구사항이 제대로 이해되지 않은 상태에서 테스트 케이스를 작성하게 되면 추후에 기능 개발이 완료된 이후에 다시 수정해야하는 일이 생길 수 있다. (나는 완벽하지 않다라고 생각해야한다.) 그렇기 때문에 개발자는 개발 초기부터 기능에 대한 요구사항을 명확히 이해하는 것이 필요하다. 이후 테스트 케이스를 작성하고 이렇게 요구사항을 정확히 이해한 후 작성한 테스트 케이스는 정말 중요한 부분이며 테스트 주도 개발이 주는 이점이라 할 수 있다.

사실 TDD의 이점은 이뿐만 아니다. 우선 잘 작성된 테스트 코드가 있고 작성된 테스트를 잘 수행하는 코드가 있다고 하면 리팩토링할 때 굉장한 이점을 준다. 사실 "좋은 코드"를 작성하기란 상당히 어려운데, 그 이유 중 하나는 코드 작성시 고려해야할 사항을 모두 인지하기가 어렵기 때문이다. 예를 들어 잘 작동하고 있는 코드더라도 유지 보수를 위해 리팩토링을 해야한다면, 개발자는 순간 숨이 턱 막히는 기분이 들 것이다. (코드에 대한 이해가 100%가 아닌 이상) 그렇기 때문에 TDD 개발이 여기서 주는 이점이 있는데, 바로 먼저 잘 작성된 테스트 케이스의 유무이다. 테스트 케이스는 코드가 변조되거나 수정되더라도 문제점을 바로 확인 할 수 있는 이점을 제공하기에 코드 퀄리티 개선을 위한 리팩토링 시 장점이 된다.
